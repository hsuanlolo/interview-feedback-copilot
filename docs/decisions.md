# Architecture Decision Records

Each decision records: context, options considered, decision made, and consequences.

## ADR-001: No hire/no-hire recommendation

**Date:** 2024-06  
**Status:** Decided

**Context:** Stakeholders initially asked for an "AI recommendation" at the top of the report.

**Options:**
- A: Include a hire/no-hire recommendation generated by the LLM
- B: Include a numeric score aggregated from interviewer ratings
- C: No recommendation or aggregate score

**Decision:** C — no recommendation or aggregate score.

**Reasons:**
1. An aggregate score has no sound statistical basis: interviewer scales aren't calibrated, anchors differ, and the sample size per candidate is 4–6 observations.
2. A recommendation compresses away the evidence that makes a decision defensible. The whole point is to surface that evidence.
3. Reproduces "AI scoring" concerns already raised by stakeholders. Trust requires transparency.

**Consequences:** Reviewers must make the final judgment. This is the intended behavior.

## ADR-002: Pydantic validation gates all LLM output

**Date:** 2024-06  
**Status:** Decided

**Context:** LLMs sometimes return partial JSON, misformatted fields, or claims without citations.

**Options:**
- A: Accept LLM output as-is, display whatever comes back
- B: Validate with Pydantic; surface errors if validation fails
- C: Validate; silently drop invalid fields

**Decision:** B — validate and surface errors.

**Reasons:**
1. Silent failure is the worst outcome in a high-stakes workflow. The reviewer must know when the extraction failed.
2. Pydantic v2 provides precise error messages that help debug prompt issues.
3. Validation forces the prompt to improve — if it keeps failing, fix the prompt, not the validator.

**Consequences:** Some LLM responses will be rejected and shown as errors. This is correct behavior.

## ADR-003: Abstraction layer for LLM client

**Date:** 2024-06  
**Status:** Decided

**Context:** We want tests to run without a real API key.

**Options:**
- A: Call Anthropic API directly from services
- B: Create `LLMClient` interface with a mock implementation for tests

**Decision:** B — abstract the LLM client.

**Reasons:**
1. Tests must run without network calls or API costs.
2. The abstraction makes it easy to swap providers (Claude → GPT-4 → fine-tuned model).
3. Mock responses can be crafted to test specific edge cases (malformed JSON, unsupported claims, etc).

**Consequences:** Slightly more code upfront. Worth it.

## ADR-004: SQLite for local development, Postgres path for production

**Date:** 2024-06  
**Status:** Decided

**Context:** We need persistence, but want zero-config local setup.

**Options:**
- A: Postgres from day one
- B: In-memory only
- C: SQLite locally, Postgres in production via same SQLAlchemy interface

**Decision:** C.

**Reasons:**
1. Portfolio demos should run with one command. Postgres requires a running service.
2. SQLAlchemy abstracts the dialect. Changing the connection string is the only required migration step.
3. SQLite is reliable enough for single-user local workloads.

**Consequences:** Some Postgres-specific features (e.g., JSON columns, full-text search) unavailable locally. Acceptable for v1.

## ADR-005: Evidence spans require exact quoted text

**Date:** 2024-06  
**Status:** Decided

**Context:** LLMs tend to paraphrase source text rather than quote it exactly.

**Options:**
- A: Accept paraphrases; verify by semantic similarity
- B: Require exact `quoted_text` that appears verbatim in the source debrief

**Decision:** B — require exact quoted text.

**Reasons:**
1. Exact matching is deterministic and auditable. Semantic similarity introduces a second LLM judgment.
2. The interviewer's exact words carry legal weight in hiring decisions. Paraphrases can distort meaning.
3. Exact quotes enable reliable character-offset verification.

**Consequences:** LLM prompt must explicitly instruct the model to quote verbatim. Paraphrases rejected.

## ADR-006: Baseline extractor before LLM extractor

**Date:** 2024-06  
**Status:** Decided

**Context:** We want to demonstrate that the LLM adds value over a simple rule-based approach.

**Options:**
- A: Build only the LLM extractor
- B: Build both baseline (deterministic) and LLM extractor

**Decision:** B.

**Reasons:**
1. The baseline gives us a lower bound on performance. LLM must beat it to justify cost/complexity.
2. The baseline works without API keys — essential for demos and tests.
3. In production, the baseline could serve as a fallback if the LLM call fails.

**Consequences:** More code, but better engineering story and eval foundation.
